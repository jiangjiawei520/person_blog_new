## 🚫 GitHub 文件大小限制说明

- **硬性限制**：GitHub 仓库**禁止**推送任何大于 **100 MB** 的单个文件。
- **警告限制**：大于 **50 MB** 的文件会收到警告，因为推荐使用 Git LFS。
- **历史提交中的大文件**：如果你的仓库历史中**曾经提交过**大文件（即使后来删除了），这个“历史记录”本身也会导致推送失败。解决这类问题需要使用特殊命令清理历史。

提供一个从零开始的、完整的操作步骤指南。请严格按照顺序执行：

### 🧰 第一步：安装 Git LFS

这是前提，你的本地环境必须安装 LFS 客户端。

1. **访问官网**：打开 [git-lfs.com](https://git-lfs.com/)。

2. **下载安装**：

   - **Windows**：下载 `.exe` 安装程序并运行。
   - **macOS**：可以使用 Homebrew 安装：`brew install git-lfs`
   - **Linux**：请参照官网的包管理器命令。

3. **验证安装**：打开终端（或 Git Bash），运行以下命令，应该能看到版本号：

   bash

   ```
   git lfs version
   ```

   

### 🔧 第二步：在你的项目仓库中启用并配置 Git LFS

假设你已经在项目目录下（即 `git clone` 下来的那个文件夹）。

1. **初始化 LFS**（在该项目仓库中）：

   bash

   ```
   git lfs install
   ```

   

   *注意：这只需在每个仓库中执行一次，它会在本地仓库的Git配置中添加一些钩子。*

2. **告诉 LFS 你要跟踪哪些大文件**。
   这是最关键的一步。你需要用 `git lfs track` 命令指定模式。

   - **如果你想跟踪特定类型的文件**（例如所有 `.zip` 文件）：

     bash

     ```
     git lfs track "*.zip"
     ```

     

   - **如果你想跟踪特定目录下的所有文件**（例如 `data/` 目录）：

     bash

     ```
     git lfs track "data/**"
     ```

     

   - **如果你知道具体是哪个大文件**（例如 `huge_model.pth`）：

     bash

     ```
     git lfs track "huge_model.pth"
     ```

     

   - （**请将上面的示例模式替换为你实际的大文件模式**）

3. **检查跟踪规则**：
   执行上一步后，会自动生成或修改一个名为 **`.gitattributes`** 的文件。你可以查看它的内容来确认规则：

   bash

   ```
   cat .gitattributes
   ```

   

   你应该能看到类似这样的一行：

   text

   ```
   *.zip filter=lfs diff=lfs merge=lfs -text
   ```

   

### 📝 第三步：提交 LFS 配置和你的大文件

现在，你需要把 LFS 的配置文件**和**你的大文件一起提交。

1. **添加并提交 `.gitattributes` 文件**：

   bash

   ```
   git add .gitattributes
   git commit -m “配置 Git LFS 用于跟踪大文件”
   ```

   

2. **添加并提交你的大文件**：

   bash

   ```
   # 添加你的大文件（假设你用的是通用的 add .）
   git add .
   # 或者单独添加 git add your_large_file.zip
   
   git commit -m “添加项目资源文件 [使用LFS]”
   ```

   

### 🚀 第四步：推送到远程仓库（GitHub）

现在，你可以像平常一样推送了。Git LFS 会自动拦截你标记的大文件，并处理上传到LFS服务器。

bash

```
git push origin main
# 或 git push origin master，取决于你的主分支名
```



**首次推送 LFS 文件时可能会比较慢**，因为需要上传大文件本身，请耐心等待。你会看到类似 `Uploading LFS objects: 100% (1/1), 158 MB | 0 B/s` 的上传进度。

### ✅ 第五步：验证

推送完成后，你可以通过以下方式验证是否成功：

1. **在 GitHub 网页上查看文件**：在仓库中点击那个大文件，你会看到文件旁边有一个 **“Stored with Git LFS”** 的标记。

2. **在本地使用命令查看**：

   bash

   ```
   git lfs ls-files
   ```

   

   这个命令会列出所有被 LFS 跟踪的文件列表。

### ⚠️ 重要提示

- **协作项目**：如果其他人要克隆或拉取你的仓库，他们**不需要**预先安装 Git LFS。但在克隆后，他们需要运行一次 `git lfs install`（不重复第一步，但需要执行这个初始化命令）才能正确下载 LFS 文件。否则他们只能看到文本指针文件。
- **`.gitattributes` 文件是关键**：这个文件必须提交到仓库中，这样所有协作者和GitHub才能知道哪些文件由LFS管理。
- **LFS 配额**：GitHub 为免费账户和付费账户提供不同的 **LFS 存储和带宽配额**。如果项目非常大，请注意用量。你可以在仓库设置的 “Git LFS” 部分查看用量。

## 方案一：文件需要保留（使用 Git Large File Storage - LFS）

如果你的大文件是**必须的**（如数据集、模型、大型媒体文件），**Git LFS** 是官方推荐方案。它会将大文件存储在专用服务器，而在Git仓库中仅保存文本指针。

**操作步骤：**

1. **安装 Git LFS**：

   - 访问 [git-lfs.com](https://git-lfs.com/) 下载安装。
   - 安装后，在终端运行：`git lfs install`

2. **指定用 LFS 跟踪大文件**（例如跟踪所有 `.psd` 文件或具体文件名）：

   bash

   ```
   # 跟踪特定文件类型
   git lfs track "*.psd"
   # 或跟踪具体某个大文件
   git lfs track "path/to/your/large_file.zip"
   ```

   

3. **确保 `.gitattributes` 文件被提交**：

   bash

   ```
   git add .gitattributes
   git commit -m "启用 LFS 跟踪大文件"
   ```

   

4. **像平常一样添加和提交你的大文件**：

   bash

   ```
   git add large_file.zip
   git commit -m "添加设计资源文件"
   ```

   

5. **推送**：现在可以正常 `git push` 了。首次推送 LFS 文件可能会耗时较长。

#### 方案二：文件不需要保留（从历史中彻底移除）

如果这个大文件是**误提交**的（如日志、编译产物、临时文件），你需要将它从Git历史中完全清除，这是一个**破坏性操作**，如果仓库是共享的，需要通知协作伙伴。

**标准操作步骤（使用 `git filter-repo` 工具，推荐）：**

1. **安装 `git-filter-repo`**：

   bash

   ```
   # 使用 pip 安装（需先安装 Python）
   pip install git-filter-repo
   ```

   

2. **备份你的仓库**（非常重要！）。

3. **运行命令移除文件历史**（将 `path/to/your/large_file.zip` 替换为你的大文件路径）：

   bash

   ```
   git filter-repo --force --invert-paths --path path/to/your/large_file.zip
   ```

   

   这个命令会重写整个提交历史，永久删除该文件的所有痕迹。

4. **强制推送到远程仓库**（会覆盖历史）：

   bash

   ```
   git push origin --force --all
   git push origin --force --tags
   ```

   

### 📌 关键建议

1. **事前检查**：在 `git add` 前，可用 `ls -lh` 命令查看文件大小。
2. **忽略不应提交的文件**：将诸如 `*.zip`, `*.log`, `node_modules/`, `*.iso` 等添加到 **`.gitignore`** 文件，从源头避免问题。
3. **处理已提交到历史的大文件**：如果错误很早以前就发生了，**方案二**是唯一选择。GitHub 官方也有帮助文档，搜索 “removing sensitive data from a repository” 可找到指南。

### ⚠️ 特别注意

- **强制推送 (`git push --force`)** 会覆盖远程历史。如果仓库有其他人协作，必须提前协调，否则会导致他们的版本混乱。
- 对于**方案二**，如果仓库是公开的且大文件已推送，即使清理了历史，文件可能仍存在于GitHub的缓存中。最彻底的做法是联系GitHub支持，或考虑将仓库设为私有后清理，再重新公开。

**请先确认你的需求**：这个大文件是项目必需的吗？如果是，走**方案一（LFS）**；如果不是，走**方案二（清理历史）**。如果需要更具体的命令指导，请告诉我文件类型和你是否需要保留它。
