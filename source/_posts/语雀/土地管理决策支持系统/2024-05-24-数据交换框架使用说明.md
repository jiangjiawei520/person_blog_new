# 1. 背景
在土地决策的开发用地预审业务时，会涉及与交换中心（后改名为：政务平台），需要进行数据采集。当时正面临政数局数据采集的版本升级，由数据采集3.0版本升级为数据采集4.0版本。最开始对接数据采集的是业务审批系统，业务要求是历史旧项目，仍然以3.0版本规范推送数据，新项目则以4.0版本规范推送数据。当时以参考借鉴的心态，查看业务审批系统的代码实现，发现由于都是纯代码实现，逻辑显得较为混乱，存在各种if判断，如果不熟悉的开发人员运维，将会有巨大的难度。另一方面，与第三方交互，一般都会提前定义接口规范，包括数据采集，相对都是比较固定的。
因此，为方便与第三方的数据交互，实现数据交换的可配置化，降低后期运维难度，考虑实现一套数据交换框架。

如果熟悉HTTP原理，并且熟练使用postman等类似工具的，会比较清楚，一般与第三方交互，在定义好接口规范之后，甚至都不需要开发，使用postman手动就能先与对方完成一系列的接口对接测试，非常方便。
一般只需要准备：请求URL，请求头（权限校验等），请求参数等，发起请求后，便可获取对方的返回结果。

那么，数据交换框架，要实现可配置化，其实最重要的实现就是获取数据，组装数据，最终形成请求参数，其它均可直接交由代码来实现。如果再细分，就如图2-2所示，当需要进行数据交换时（数据交换框架，目前更多的定位是在推送数据至第三方），都需要一个触发的事件。例如：流程环节提交，推送数据至政务平台，等等。触发事件需要指定一个推送的类型，声明这个推送类型的被称为“任务类型”，特指一种概括性的推送类型。任务类型下，会授权绑定一个或者多个任务，任务是具体的，对应上述HTTP原理时的每一个请求，毕竟一个推送类型下，可能需要发送多个请求来完成。任务下，又会授权绑定一个或者多个的任务接口，其主要作用是查询数据库，为任务提供待组装数据。
例如：任务类型--用地预审受理，其下会授权多个任务：推送受理表任务，推送受理材料目录任务，推送受理材料任务，等，而每个任务下又会绑定对应的任务接口：查询受理表任务接口，查询受理材料目录任务接口，查询受理材料接口，等；每个任务通过任务接口获取到对应的数据后，会进行数据组装，最后发起请求，完成数据推送。

### 3.1.1. 具体任务类型
![image.png](https://cdn.nlark.com/yuque/0/2023/png/3012478/1675652863548-c5d9c88c-df1f-4904-9cca-a2d6cde0bc9f.png#averageHue=%23faf1eb&clientId=u7072275b-8b9f-4&from=paste&height=875&id=zX2KV&originHeight=1444&originWidth=1123&originalType=binary&ratio=1&rotation=0&showTitle=false&size=218567&status=done&style=stroke&taskId=ub61c00f7-004c-4388-86b2-726403536a1&title=&width=680.606021268092)
图3-1-2
具体任务类型，是指实际真实推送的任务类型。
任务交换目标：目前有三种：统一申办交换中心（已改名为：政务平台）、一网通办、部设施农用地，都是明确具体请求的第三方。根据公司领导的战略设计要求，凡是对外交换的，都需要先通过政务平台，再由平台发送请求至第三方。因此，后续如果有新的交互需求，都需要考虑先直接与平台交互，再由平台与第三方交互。
是否存在不交换情况：主要是用于确定该具体任务类型是否需要推送，如果选择是，就需要配置是否交换任务接口，当任务接口返回的参数is_exchange=1，则交换，若is_exchange=0，则不交换。如果默认必须交换，则可以直接设置为否即可。例如：用地预审的双公示的数据推送，当项目中确实设计双公示时才需要推送，否则不需要推送，此类业务场景，就需要使用到此配置。
是否查询新参数：默认已有参数：（1）业务参数：项目ID（xmid），申办流水号（sblsh）等；（2）流程参数：流程实例ID（processinstid），活动实例ID（activityinstid），工作项ID（workitemid）等，如果这些默认参数不能满足实际需求，则可以选择是，绑定任务接口，查询返回的结果将作为新增参数。注：此类参数，在配置引用时，不区分大小写。
调用方式：同步，表示直接触发推送数据，推送失败会限制业务（流程）往下执行；异步，表示间接触发推送数据（目前是通过定时器的方式，批量推送），推送失败不会限制业务（流程）往下执行。
是否分组：应用于任务类型之间执行先后顺序的控制，对于同一个项目，会根据任务类型的任务分组序号升序推送数据，如果排序在前的任务类型未推送成功，排序在后的任务类型将限制推送！
特别注意：不同的业务，应该设置不同的分组，避免相互影响！！！
任务类型版本：默认是“1”，如果存在不同的版本，那么通常只能启用其中之一。

操作内容：对应HTTP请求中的请求参数，目前只支持JSON数据格式。形如：
```json
{
	"XXX1": #query_table.id#,
	"XXX2": #query_table.name#,
	"XXX3": $ydys_ywtb_accept_add_attach_v4:query_ydys_ywtb_accept_fjmlxxb.mat_id=query_ydys_ywtb_accept_fjxxb.material_catalog_id$
}
```
其中参数值，通常是以“##”或者“$$”符号包裹。“##”表示普通参数引用，通常是以“#任务接口编码.参数编码#”的形式。“$$”表示引用其它任务，通常是以“$其它任务编码$”或者“$其它任务编码:当前任务的任务接口编码.参数编码=其它任务的任务接口编码.参数编码$”的形式，主要是用于JSON参数的对象嵌套，理论上是可以嵌套多层的，目前最多只嵌套2层，例如：当需要推送材料数据时，第1层是嵌套材料目录，第2层是嵌套材料参数；若需要先判空，支持嵌入判空方法，例如：“$isEmpty(query_ydys_ywtb_finish_fjmlxxb, ydys_ywtb_finish_add_material_v4)$”，若参数1中的任务接口返回结果为空，则直接替换成“null”。当然，如果单纯的参数引用无法满足需求，此处还支持对参数的方法处理：

# 4. 注意事项
## 4.1. 未成功交换任务过多
造成此问题的主要原因有两个：
（1）为不影响流程往下流转，任务类型的调用方式配置为“异步”，目前的实现是通过普元定时器，根据数据交换任务表（ex_exchange_task）记录，触发任务交换，默认是未交换状态；
（2）由于系统本身原因，或配置错误，或数据问题，也有可能是外部原因，导致推送失败；
由于定时器是批量推送未成功的任务，包含未交换和交换失败，需要经常监管，及时处理，避免越积越多，难以处理，很多数据推送，会有时间限制，例如：一网通办的数据交换，当时业务上是要求，从用户操作，到数据推送，整个过程不超过5分钟，如果推送过程出现异常，很容易就超过该时限要求。

此问题，主要针对用地预审省级业务，应用定时器从政务平台接入新项目。定时器设置如图4-2-1所示，其中有一个参数，是指定每次接入的项目数量（count），根据项目申报时间倒序获取，如果存在倒序前count个项目都接入失败，则会导致无法接入新的项目，这个需要特别注意，需要定时监控。目前做了优化处理，记录接入项目次数，如果超过一定次数（目前默认是5次）仍然无法正常接入，则定时器批量获取项目时，暂时不再获取此类项目，待运维人员解决问题后，再在接收数据日志功能中手动接收，手动接收不做次数限制。
## 4.3. 流程未正常触发新增交换任务
流程流转时，有时候会出现不明原因的无法正常触发流程事件，并新增交换任务，运维时需要手动重新新增交换任务，步骤如下：

1. 查询获取该流程活动环节的工作项ID；

一般需要推送数据的都是在主流程活动环节，如果是属于主流程，可以根据业务ID（ywid），利用如下SQL获取工作项ID：
```sql
select wfw.*
  from wfprocessinst wfp
  left join wfworkitem wfw
    on wfp.processinstid = wfw.processinstid
 where wfp.extend2 = '业务ID'
 order by wfw.createtime desc;
```
如果类似用地预审与规划许可业务，办结环节是会发起子流程，推送数据是在子流程的环节中触发推送，可以根据业务ID（ywid）和活动环节名称（activityinstname），利用如下SQL获取工作项ID：
```sql
select wfw.*
  from wfprocessinst wfp
  left join wfactivityinst wfa
    on wfp.processinstid = wfa.processinstid
  left join wfworkitem wfw
    on wfa.subprocessid = wfw.processinstid
 where wfp.extend2 = '业务ID'
   and wfa.activityinstname = '活动名称'
 order by wfw.createtime desc;
```
其中的workitemid字段，即为工作项ID，请特别注意看工作项名称字段（workitemname）避免搞错。如果同一个活动环节，有多条工作项记录，则需要根据创建时间（createtime）先后顺序来判断。

2. 登录该用户账号，新增交换任务时需要记录该用户信息：

此处需要特别注意，如果是在当前浏览器的其他页面登录其他用户的，需要使用该项目该流程活动环节的用户重新登录，保证用户信息的正确性。

3. 在浏览器打开一个新的标签页，并在地址栏填入URL，回车发起请求：

测试环境：
http://172.16.101.19:8080/default/com.geostar.gtgh.jsydsp.gdgz.common.sysmgr.task.exchangeTask.addExchangeTask2.biz.ext?ywid=&exType=&workItemID=&sysMgr=false
正式环境：
http://19.16.240.43:8080/default/default/com.geostar.gtgh.jsydsp.gdgz.common.sysmgr.task.exchangeTask.addExchangeTask2.biz.ext?ywid=&exType=&workItemID=&sysMgr=false
需要在等于号“=”后面，分别补充完善业务ID（ywid）、交换类型（exType，即任务类型）、工作项ID（workItemID），其中默认sysMgr=false，指明获取当前用户信息，否则如果sysMgr=true，则表示获取系统管理员为当前用户信息。
注意：此处是打开一个新的标签页，不是打开一个新的窗口，不然将无法获取当前登录的用户信息。

4. 如果成功，在交换任务管理功能中，将能看到该记录信息。
