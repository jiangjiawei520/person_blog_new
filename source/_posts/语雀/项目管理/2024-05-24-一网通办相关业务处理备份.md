## 一、解决一表申办与交换中心事项目录数据不一致的临时方案

**1、交换日志报错提示一般分以下几种情况：**


解决：由于一表申办与交换中心省事项目录唯一ID不一致导致无法查询到事项目录信息的临时方案：

<!--more-->
1. 在土地决策数据库，查询地方实施编码

select local_Task_Code, item_row_id from ydys_ywtb_sbxxb where ywid = '';

2. 在交换中心MySQL数据库，根据地方实施编码，查询最新版本省事项目录唯一ID

select ROWGUID from audit_item_v2 where task_code = '' order by TASK_VERSION desc LIMIT 1;

3. 更新土地决策数据库，省事项目录唯一ID

update ydys_ywtb_sbxxb set item_row_id = '' where ywid = '';
根据和数广负责人王世峰沟通，现在这个事项目录不一致的问题，有了解决方案。之前我们是根据一表申办通过7.1接口中的省事项目录唯一ID查询交换中心的事项目录信息，相当于强关联了一表申办和交换中心的版本必须是一致的（双方的事项目录信息来源都是一致的），才能在交换中心查询到数据。王总那边的说法是：事项目录本身设计是异步的，一表申办和交换中心并不需要完全一致，每个系统只需要关注本系统所拉取的事项目录信息中的当前最新版本。我们需要进行改造：每次只根据地方实施编码，查询交换中心最新版本的事项目录信息，这样就可以避免类似查询不到事项目录信息的问题了。

### 1、规划许可

1. 查询所有建设用地规划许可的交换任务

## 六、7.1接口附件下载异步处理说明
由于省中台对接口总时长统一限定90秒。附件下载相对用时较长，故使用异步处理的形式，避免请求超时。异步处理，不会占用接口请求时长，但是会存在缺陷：如果附件下载接口请求异常，附件无法正常下载，然而7.1接口已经返回接收成功信息，是存在前后矛盾的。当然接口正常，此类情况发生概率相对较小，一旦出现需要手动处理，重新下载附件。

## 十、用地预审省级业务数据流转说明

1. 进件入口：统一申办
2. 接收方式：统一申办申请提交后，交换中心通过ETL方案把数据抽取至交换中心（具体实现方式可以忽略，后续可能会改造优化，取代ETL方案的方式），然后由土地决策实现的定时任务主动抽取交换中心数据至业务系统。
3. 推送方式：土地决策系统推送办件过程数据、行政双公示结果至交换中心，再由交换中心把数据推送至广东省大数据平台中间库（也称： 省大数据中台，由 省政数局负责）。另外， 效能监察、好差评、双公示等会从广东省大数据平台中间库获取相应的数据。
4. 目前成果：已经实现上述整个业务流转的功能，但是目前只是内部测试过完整业务流程，也给自然资源厅相关领导演示过，暂时没有正式推广使用。
